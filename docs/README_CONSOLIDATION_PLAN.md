# README Consolidation Plan

## Purpose

This plan prepares README consolidation for Future of ID before any README edits, file moves, deletions, or archive decisions happen.

It compares the current README files against the documentation architecture, inventory, and docs hub so the final README strategy can be deliberate instead of piecemeal.

## Current README Inventory

### `README.md`

**Current role**

The current root README is the strongest active public-facing project overview. It explains the product, audience, learning model, seven capability areas, philosophy, current state, stack, repository structure, local development, public repo rationale, license, and author context.

**Strengths**

- Strongest current explanation of what Future of ID is and why it exists.
- Uses the current product-facing term `Reverse Build`.
- Clearly frames the project as both a professional learning experience and a transparent reference implementation.
- Includes useful audience, capability, current-state, stack, repository, local development, license, and public-repo rationale sections.
- Best candidate foundation for the final root README.

**Weaknesses**

- Too long for a focused public front door.
- Mixes public positioning with architecture/documentation inventory detail.
- Repository documentation list is likely to drift as docs reorganize.
- Current-state section may become stale quickly.
- Formatting is plain and could be tightened into a more scannable root README.

**Recommended future role**

Keep as the base for the final root `README.md`, but trim it into a concise public-facing entry point that links to `docs/README.md` for deeper documentation.

### `READMEV2.md`

**Current role**

An earlier concise project overview focused on vision, audience, course theme, seven capabilities, experience model, proposed stack, and why the project exists.

**Strengths**

- Clear, compact statement of project vision.
- Strong line: “The course teaches the future of instructional design by using the future of instructional design.”
- Useful audience framing.
- Useful seven-capability list.
- Concise experience model and stack summary.

**Weaknesses**

- Uses older “course” framing more heavily than current platform/guided-course direction.
- Uses “reverse engineering” in the experience model where product-facing language should now prefer `Reverse Build`.
- Proposed stack language may be partially superseded by the current implementation and docs.
- Too thin to serve as the final root README by itself.

**Recommended future role**

Merge the strongest positioning, audience, and course-theme language into the final root README or strategy docs. After consolidation, treat `READMEV2.md` as superseded historical material.

### `READMEV3.md`

**Current role**

A later README draft with stronger public positioning, badges, feature framing, project philosophy, tech stack, architecture exploration, local development, contributing, license, and author note.

**Strengths**

- Strong tagline: “A living demonstration of the future of instructional design — built by an instructional designer, for instructional designers.”
- Good compact feature list.
- Good explanation that the platform is both the learning experience and the reference implementation.
- Useful public-facing project philosophy.
- Includes contributor/license language that could inform the final README.

**Weaknesses**

- Uses `Reverse Engineering` as product-facing language; this should become `Reverse Build` in the final root README.
- Markdown code fence appears to swallow later sections, making formatting unreliable.
- Repeats tech stack content.
- Some phrasing is now less current than the active root README.
- Should not remain as a competing root README.

**Recommended future role**

Extract the strongest tagline, feature framing, and contributing/license wording. Then archive or remove after approved consolidation.

### `web/README.md`

**Current role**

A default Next.js app README generated by `create-next-app`.

**Strengths**

- Contains basic app-level startup and deployment references.
- Located near the actual Next.js app, which makes it a reasonable home for app-specific contributor workflow later.

**Weaknesses**

- Mostly generic Next.js boilerplate.
- References editing `app/page.tsx`, which is not useful for the current app structure.
- Does not reflect Future of ID architecture, scripts, environment variables, AI routes, or project workflow.
- Duplicates setup content that should be coordinated with the root README.

**Recommended future role**

Keep `web/README.md` as app-specific implementation guidance later, focused on scripts, local development, environment variables, app directory structure, validation commands, and deployment notes. Do not use it as the public project overview.

## Recommended Root README Direction

The final root `README.md` should become the public-facing project front door.

It should provide:

- a concise project overview
- the strongest project positioning
- who the project is for
- why the project matters
- the core learning model
- the seven capability areas, listed briefly rather than fully explained
- a short current-state snapshot
- local development instructions
- a link into `docs/README.md`
- license and attribution context

It should not become:

- a full architecture manual
- a complete documentation map
- a replacement for `docs/README.md`
- a changelog
- a long roadmap
- a duplicate of strategy, architecture, or design-system docs

## Content to Move Into `docs/README.md`

The root README should link to `docs/README.md` instead of duplicating internal documentation navigation.

Move or keep only in the docs hub:

- long documentation file lists
- documentation domain descriptions
- recommended reading paths
- source-of-truth tables
- migration cautions
- AI assistant documentation navigation
- detailed architecture navigation
- detailed decisions/validation navigation

The final root README should include a short “Documentation” section that links to `docs/README.md` and names only the most important entry points.

## Terminology Checklist Before Rewrite

Before rewriting the root README, apply this terminology pass:

- Use `Reverse Build` for product-facing language.
- Use `reverse engineering` only as an explanatory concept.
- Use `platform` for the full Future of ID product.
- Use `guided course` for the first major learning surface.
- Avoid inconsistent switching among `course`, `platform`, and `learning experience`.

## Content to Archive or Remove

Archive or remove later, after approved consolidation:

- duplicate README explanations of project purpose
- duplicate seven-capability explanations outside the final root README and source-of-truth strategy docs
- stale `Reverse Engineering` product-facing terminology in README drafts
- generic Next.js boilerplate in `web/README.md`
- stale setup instructions that do not match the actual app workflow
- overly long architecture exploration sections in the root README
- duplicated tech stack sections
- broken or malformed Markdown from `READMEV3.md`
- README variants once their strongest content has been preserved

Do not delete `READMEV2.md` or `READMEV3.md` until their useful language has been merged or intentionally preserved elsewhere.

The seven capabilities should be listed in the root README, not fully explained there. Deeper rationale belongs in `docs/strategy/FUTURE_READY_CAPABILITY_MODEL.md` and related strategy docs.

Project status should stay lightweight. Current-state content should be brief and link to roadmap or documentation rather than becoming a changelog.

## Proposed Root README Outline

Recommended final `README.md` outline:

1. `# future_of_id`
2. Short tagline
3. One-paragraph project overview
4. Why this project exists
5. What learners experience
6. Core learning model: `Theory → Experience → Reverse Build → Reflection`
7. Seven capability areas
8. Who this is for
9. Current project status
10. Documentation / where to learn more
11. Tech stack
12. Local development
13. Contributing / questions
14. License
15. Built by Bradley Pierce

The root README should stay readable in one pass. Detailed architecture, design system, AI, validation, and governance links should point into `docs/README.md`.

## Proposed `web/README.md` Role

`web/README.md` should remain app-specific if retained.

Recommended future scope:

- app-specific setup from inside `web/`
- available npm scripts
- environment variables
- local development workflow
- validation commands
- app directory structure

- troubleshooting notes for the Next.js app

It should not explain the full product vision, documentation architecture, or learning model except as brief context with links back to the root README and `docs/README.md`.

## Risks to Avoid

- Making the root README too long.
- Burying the project value proposition below setup or architecture details.
- Keeping competing README versions too long after consolidation.
- Breaking links when README variants are later archived or removed.
- Duplicating `docs/README.md` inside the root README.
- Losing useful language from `READMEV2.md` or `READMEV3.md`.
- Leaving stale `Reverse Engineering` product-facing terminology in public-facing README content.
- Preserving generic `create-next-app` boilerplate as if it were project-specific guidance.
- Allowing setup instructions to drift between root `README.md` and `web/README.md`.
