# Future of ID — Reverse Build Experience

## Purpose

Reverse Build helps learners inspect the learning experience they just completed from the instructional designer's perspective.

It exists to make professional reasoning visible. Learners should not only complete an activity, receive feedback, and move on. They should be able to pause, inspect the design choices behind that experience, and understand how instructional judgment shaped what they just did.

Reverse Build is a signature Future of ID learning experience. It is not a generic design-notes section, developer explanation, or implementation commentary.

## Core Definition

As an experience principle, Reverse Build reconstructs the learning experience from the instructional designer's perspective. It does not reverse build the module, the topic, or the code. It reveals the intentional thinking, instructional foundations, design decisions, AI collaboration, trade-offs, and learning science that shaped the experience.

The current implementation presents authored professional reasoning and does not yet receive or display the learner's actual Design Practice response or AI Review. Reverse Build v2 will add that learner-artifact comparison only after the approved implementation brief, shared artifact architecture, reference implementation, and validation requirements are completed.

The target learner-facing signature for Reverse Build v2 is:

"You just experienced it as a learner. Now let's inspect it as an instructional designer."

## The Apprentice Principle

Reverse Build should feel like standing beside an experienced instructional designer while they reconstruct their thinking aloud.

The learner is not being shown the "correct answer."

They are being invited into the professional reasoning process that shaped the experience.

The goal is mentorship rather than explanation.

## What Reverse Build Is Not

Reverse Build is not:

- a summary of the module topic
- a developer walkthrough
- a code explanation
- generic design notes
- a post-lesson recap
- a required linear prerequisite

## The Four Design Questions

### 1. What are we reverse building?

We are reconstructing the learning experience the learner just completed from the instructional designer's perspective.

Not the module.

Not the topic.

Not the code.

The learning experience.

### 2. What should the learner see first?

In Reverse Build v2, the learner should first reconnect with available evidence from the learning experience before instructional reasoning is revealed.

The v2 experience should begin by establishing the object being inspected:

"You just experienced this as a learner. Now let's inspect it as an instructional designer."

In the Reverse Build v2 target, the learner should see a concise reconstruction using available evidence:

- the learning challenge
- the Design Practice decision or response, when available
- the associated AI Review, when available

Reflection remains a separate activity after Reverse Build. Reverse Build must not receive, display, require, summarize, or depend on the learner's Reflection response or feedback.

Then Reverse Build progressively reveals:

- the problem
- instructional foundations
- design decisions
- AI collaboration
- design trade-offs
- learning science and transfer

Reverse Build should reveal what is new by connecting it to what instructional designers already know.

### 3. What should the learner discover?

The learner should discover that great learning experiences are intentionally engineered through thoughtful instructional design decisions, not accidentally assembled or generated by AI.

They should realize:

- there is usually more than one reasonable design solution
- every instructional decision involves trade-offs
- AI can accelerate thinking but does not replace instructional judgment
- established instructional design principles continue to matter
- professional reasoning can be observed, understood, and developed

Reverse Build does not teach learners what to build. It teaches them how to think about building.

### 4. What should the learner believe after Reverse Build?

The learner should believe that professional judgment is learnable.

They should no longer see effective learning experiences as finished products to imitate. Instead, they should see them as collections of thoughtful design decisions that can be studied, questioned, adapted, and eventually applied in their own work.

"I can learn to think like the person who built that."

## Reverse Build Layers

### 1. The Experience

In the Reverse Build v2 target, reconnect the learner to the activity using the challenge, available Design Practice response or decision, and available AI Review. This anchors Reverse Build in observable practice rather than abstract explanation. Reflection remains the subsequent transfer activity and is not part of the reconstructed input.

### 2. The Problem

Name the instructional or performance problem the experience was designed to address. The learner should understand why this experience existed before seeing how it was built.

### 3. Instructional Foundations

Connect the experience to established instructional design and learning science foundations. Reverse Build may naturally reference ADDIE, Backward Design, performance consulting, cognitive load, deliberate practice, retrieval practice, or other relevant models.

The goal is not to reteach these models. The goal is to show how modern instructional design builds on them.

### 4. Design Decisions

Reveal the major choices that shaped the learner experience: what was included, what was simplified, what was sequenced first, and what kind of learner judgment the activity was meant to develop.

### 5. AI Collaboration

Show where AI supported the design process, such as drafting, variation, feedback, critique, or scenario development. Also show where human judgment reviewed, refined, overrode, or rejected AI suggestions.

### 6. Design Trade-offs

Make the constraints visible. Reverse Build should show that design decisions involve trade-offs between clarity and realism, depth and usability, speed and quality, guidance and autonomy, or automation and human judgment.

### 7. Learning Science & Transfer

Explain how the experience supports transfer beyond the page. This layer should help learners connect what they practiced to future workplace design decisions.

## Experience Principles

- Reverse Build must be anchored to the experience the learner just completed.
- Every Reverse Build must stand on its own.
- Reverse Build should not assume the learner completed prior modules.
- Reverse Build should connect new practices to established instructional design foundations.
- Reverse Build should show AI collaboration honestly, including where human judgment overrode, refined, or rejected AI suggestions.
- Reverse Build should highlight trade-offs, not pretend there was one perfect answer.
- Reverse Build should feel like mentorship, not documentation.
- Reverse Build should support open exploration and not imply a required sequence.

## MVP Guidance

The current implementation presents authored reasoning about the React-based module experience. It does not yet reconstruct learner-specific Design Practice or AI Review artifacts.

The approved Reverse Build v2 target will reconnect available Design Practice and AI Review artifacts with the authored reasoning while preserving open access and keeping Reflection separate. `REVERSE_BUILD_V2_IMPLEMENTATION_BRIEF.md` controls v2 inputs, states, privacy, accessibility, technical boundaries, claims, and acceptance criteria after Bradley approves it.

In future versions, if Storyline, iSpring, or richer interactive learning experiences appear at the top of each module, Reverse Build should naturally reconstruct those experiences.

The core concept does not change: the object being reverse built is always the learning experience the learner just completed.

## Design Constraints

Reverse Build should reveal instructional thinking.

It should not overwhelm learners with implementation detail.

Reverse Build generally avoids exposing:

- React implementation
- Storyline variables
- JavaScript logic
- CSS decisions
- API architecture

unless those implementation choices directly contribute to instructional reasoning.

The learner should leave understanding better instructional design—not better source code.

## Implementation Guardrails

- Do not present Reverse Build as optional footnotes.
- Do not bury Reverse Build behind vague "design notes" language.
- Do not turn Reverse Build into a code walkthrough.
- Do not over-explain learning theory.
- Do not make Reverse Build dependent on module order.
- Do not make Reverse Build receive, display, require, summarize, or depend on Reflection.
- Do not imply AI made the design decisions.
- Use learner-facing language such as "Inspect the Build," "Reveal the Thinking," or "Reconstruct the Experience."
